## What CC10 built

### 1. The janitor (`pipeline/janitor.py`)
ONE manifest names every Prisma model with a policy — `Forever` (the record), `Replace` (the
self-cleaning forward calendar), or `Trailing(N days|sessions, column)`. A unit test mirrors the
manifest against `schema.prisma` in BOTH directions, so a new table with no policy — or a policy for a
table that no longer exists — reds the build (the routes-manifest guarantee, applied to the data
model). The DELETE targets derive from the `Trailing` entries ONLY, and `_guard` refuses any table off
that derived allow-list — a test proves a `forever` table (briefing, signal_log, watchlist_item)
raises. signal_log / signal_resolution keep their DB insert-only triggers as a second, physical lock.

The stage runs in the full nightly, after publish and before revalidate. News deletes in FK-safe
order — child links, then clusters, then articles (floored at the snapshot cutover), then any image no
surviving cluster still references. Sessions-tables trim to a trading-session floor (30 sessions);
price bars keep 400 sessions (the R2 parquet lake, which compute mode reads, is untouched). The one R2
deletion trims the `backups/` prefix to the most recent 8 dumps. `publish_janitor` stamps a `janitor`
entry beside the night's `source_status` — the exact mirror of `publish_dawn`, merged with `||` so a
healthy night never reads degraded because deletion ran.

**Manifest quirks, ruled and marked:** `watchlist_item` → `forever` (the reader's own data — the one
model the plan's §4.8 table did not name, Q-CC10-1). `catalyst_link` has no date column of its own, so
it is `TrailingByParent(news_cluster, run_date)` — retired when its parent cluster is. `news_image`
trims by `fetched_at`, but only when unreferenced (the cluster→image FK).

### 2. The citation snapshot (`briefing.sourcesJson`, one additive migration)
`publish_briefing` now freezes `[{title, outlet, url}]` of the articles a brief was written from into
the briefing row, so the janitor's news purge never orphans a published brief — its sources resolve
with no `news_item` join (a pipeline test proves it: purge the article, the brief still resolves). The
janitor reads the earliest snapshotted brief (`SELECT MIN(run_date) WHERE sources_json IS NOT NULL`) as
its news-deletion cutover: before that date a brief's sources are only joinable, so its news is never
purged. Migration `20260716194641_briefing_sources_json` — additive, one nullable JSONB column.

### 3. The control-room Janitor row (`lib/pipelines.ts` + `PipelineSheet.tsx`)
A 4th schedule row — "Janitor · Retires stale news, scans and old backups on the nightly run — the
record is kept forever." Its sheet reports last night's retirements in the mechanical voice ("Retired
last night: 214 news items past 45d · 1 sessions of scan rows · backups kept 8"), read from the
`janitor` entry via `readJanitorEntry`/`lastRunForJanitor`. `buildSourceStatus` and `nightSources` skip
the janitor key exactly as CC9 taught them to skip the dawn key — the footer never prints
"[object Object]" as a phantom degraded provider (the CC9 footer bug's sibling, closed in the same shape).

### 4. The "new" tags (R8) — `components/NewTag.tsx`
A cluster first seen after the prior edition's press time (`getPriorEditionPressTime` = the
2nd-most-recent finished run's `finishedAt`; null prior tags nothing) wears one quiet lowercase mono
word after its headline. Information, never urgency — no count, no badge, no colour, nothing that
moves. Edition-relative, so cache-safe and reader-untracked. On the Desk front-page module and /news;
NOT on the Morning Plan's self-labeling Overnight (that whole section is "since the close"), nor on the
story sheet (a single-story deep view is not where "what's new" is scanned). `copy.ts` owns the word,
`NewTag` is its one renderer. The calendar tag is deferred (Q-CC10-2): `calendar_event` is
replace-policy with no first-seen column, so an honest tag needs a second migration the plan does not
sanction.

**Bundle boundary, learned at the ceiling:** `isNewInEdition` lives in `lib/news.ts`, not
`lib/pipeline.ts` — the latter imports the Prisma client, and `news.ts` is pulled into the /news CLIENT
bundle by `NewsCard`. Putting the pure helper in `pipeline.ts` dragged Prisma onto /news and blew the
200 KB ceiling (216 KB). Moved; /news landed back at 199.8 KB.
