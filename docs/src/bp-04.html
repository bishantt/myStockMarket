Part 6

Data model

A deliberate split: full-market history lives as Parquet files the pipeline scans; Postgres holds what the app serves — plus the small set of tables the user writes (watchlist, journal, review state, paper trades). Two stores, each doing the one thing it is good at.

6.1  Table definitions — the fields that carry design decisions

TablePurpose & the columns that matter
instrumentSymbol master for the defined US universe (NYSE/Nasdaq/AMEX). is_active/delisted_at exist so backtests can include dead tickers — the survivorship guardrail from the Research Report is a schema feature, not a hope.
price_barThe serving copy of price data — watchlist plus index/sector symbols × ~5 years of daily bars (≈40–60k rows, trivial inside the 500MB ceiling), written nightly like every serving table. Candlestick charts and sparklines render from here; full-market history stays Parquet-only, and the app never needs R2 credentials.
base_rate_statOne row per (pattern_key, universe, horizon, regime): n, wins, win_rate, ci_low, ci_high (Wilson), fwd_p10/median/p90, plus pattern metadata: publication_year, evidence_grade (the Part-4 ledger verdict), and decay_note — the decay stamp is a schema field, not a caption. The UI’s display-precision rules key off n; the tier is capped at “weak” whenever the CI includes the always-up baseline.
vol_bandThe product’s only sanctioned forecast: per-watchlist-symbol expected-range bands at 5/10/20 trading days from empirical quantiles of recent realized moves (HAR-lite), each row carrying its frequency label (“8 in 10 past paths stayed inside”) and the regime-break caveat string. Nothing beyond 20 days exists, by schema.
setup_cardA day’s rendered signal: pattern state, tier, weakeners with live check-state (the checkbox is user-writable), FK to its base-rate row. Cards are regenerated nightly; history lives in the log.
signal_log + signal_resolutionThe track record, mechanically honest. signal_log is insert-only (UPDATE/DELETE revoked): fired signal + the exact numbers stated at fire time + resolves_on, with UNIQUE (fired_date, pattern_key, symbol, horizon) and ON CONFLICT DO NOTHING so reruns are idempotent. The nightly resolver inserts a signal_resolution row (outcome, resolved_at) — outcomes live in their own insert-only table, so append-only is a mechanical property, not a convention. Logging starts in Phase 1, resolution in Phase 4.
paper_tradeThe paper-trading ledger — v1’s only “orders.” Symbol, side, qty, simulated fill (next-open price + half-spread + slippage estimate), opened/closed timestamps, a required thesis note, optional link to the signal that prompted it. Turnover from this ledger drives the cost mirror and the trade-frequency mirror; a paper order placed within minutes of a signal firing triggers the cooling-off interstitial. This table is what makes “paper-first” a feature rather than a slogan.
briefingAM/PM JSON plus verification_json (every extracted number/ticker/date and where it matched) and model_meta (models, token counts, batch vs sync) — the audit trail for every published word.
news_itemSnippet + provider URL + the Stage-A extract. Snippets only (Marketaux never returns full text); the UI always links out.
calendar_eventUS earnings (with bmo/amc), US macro prints (consensus/prior — CPI, jobs report, GDP), FOMC dates and Fed speakers. The if/then scenario cards render from this plus base_rate_stat — and a branch whose historical sample has N < 30 renders “insufficient history for this branch — treat as anecdote” rather than a number.
pipeline_runPer-stage and per-source status — surfaces as the dashboard footer (“news unavailable tonight”) so degraded runs are visible, never silent. Stage-completion rows are what make workflow_dispatch reruns skip finished work.
lesson / concept_stateAcademy content (MDX) and the Leitner review queue (box, due_on, familiarity for adaptive fading). concept_state is app-written on every review answer.
journal_entryPM-prompt responses — plus the forecast-journal fields the Research Report commits to: optional forecast (a falsifiable statement), probability (anchored on a displayed base rate), resolves_on, outcome, so user predictions are Brier-scored on the same track-record page as the app’s own flags. Together with paper_trade, the only truly irreplaceable data in the system — see backup policy, §7.3.

Parquet layout: prices_daily/year=YYYY/*.parquet (current year rewritten nightly; prior years rewritten per-symbol when a corporate action lands), adjustments/ as the split/dividend audit trail, indicators_daily/ and signal_events/ regenerated from prices as needed. Losing the bucket costs one re-pull from Alpaca — by design, nothing in R2 is precious.

Part 7

Security, operations & cost

Single-user security is mostly about not inventing problems: one wall, few secrets, and monitoring designed around the one catastrophic failure mode — silence.

7.1  Authentication & the licensing wall

HTTP Basic Auth implemented in ~20 lines of Next.js middleware (proxy.ts in Next 16): credentials in env vars, constant-time comparison, HTTPS via Vercel. This is deliberately boring — for one user it is exactly sufficient, and it doubles as the data-licensing compliance mechanism (every provider requires the dashboard stay private to you). The pipeline’s revalidate route requires an Authorization: Bearer $CRON_SECRET header. Vercel’s free preview-deployment authentication stays on so preview URLs never expose data pages. Rejected: Vercel Password Protection (requires a $150/mo add-on — absurd here), Clerk (dependency overkill for n=1); the free step-up if Basic Auth’s browser popup ever grates is Auth.js with a single credentials user, or Cloudflare Access in front of a custom domain.

7.3  Monitoring, failure design, and backups

  • The catastrophic failure is a silently dead cron — stale data quietly served day after day. Defense in depth: a zero-cost heartbeat step in the workflow (the 60-day schedule auto-disable is documented for public repos; unverified for private — the heartbeat removes the question either way); workflow-failure e-mails catch crashes; and a healthchecks.io dead-man check configured on the pipeline’s own cron schedule (25 0 * * 2-6, ~45-minute grace) so weekends never false-alarm — absence of success alerts by ~9:10pm EDT (8:10pm EST), the same evening, not days later. The holiday-exit path pings success explicitly so US market holidays don’t false-alarm either.
  • Degraded > dead: each data source is isolated, so one provider outage turns into one “unavailable tonight” section, recorded in pipeline_run and shown in the dashboard footer. Ingest hard-fails if <95% of the expected universe returns, so stale data never masquerades as fresh.
  • Publish is one transaction: all serving tables are refreshed inside a single transaction (DELETE-and-INSERT of the night’s serving rows, or staged tables renamed together in the same transaction) — MVCC serves readers the previous consistent snapshot until commit, so the Desk never renders a half-written briefing and foreign keys never dangle across generations.
  • Backups follow value: Parquet is re-pullable and pipeline-derived Postgres tables are regenerable — but signal_log/signal_resolution, paper_trade, and journal_entry are irreplaceable, so a weekly pg_dump step (over the session-mode pooler — see §7.2) ships a compressed dump to R2. Restore is one command; testing it once is a Phase 1 acceptance criterion.
