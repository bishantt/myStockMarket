Part 1

Decisions at a glance

One deliberate departure from the familiar path, one architectural split, and a bill that rounds to a coffee. Everything else is the stack you already ship with.

1.1  The shape of the system

The app is read-mostly by construction: a Python pipeline does all the work in the cloud each evening after the close — ingest, scan, compute base rates, write the briefing — and the web app just renders the results. That single decision collapses most architectural questions: no always-on backend, no queues, no websockets on day one, no server for you to babysit. The one departure from your Vercel-native habits is forced by physics: Vercel Hobby caps every function at 300 seconds and fights pandas-sized Python bundles, while a scheduled GitHub Actions workflow in a private repo gives a full Python environment, 6-hour job limits, and free minutes — and the wide after-close window (the pipeline needs barely an hour of it before your evening read) absorbs its cron jitter with room to spare.

Part 2

System architecture

Seven boxes, one direction of flow. Everything heavy happens in two short cloud jobs in your early evening, right after the close; the app is a thin, authenticated window onto their output. Every trigger is fixed in UTC — DST-proof by construction: when your clocks shift (you live on US Eastern), the triggers don’t move, and they sit after the close in both EDT and EST.

2.1  An evening in the life of the pipeline

All triggers are fixed in UTC; your clock is US Eastern, which observes DST. So the local times below shift by an hour between summer and winter while the UTC triggers stay put — and both regimes clear the close comfortably. The design splits the evening into two short jobs so no runner ever sits idle waiting on the LLM batch.

Why the timezone math is friendly

You live on market time, so there is no conversion to reason about — the close is 4:00pm on your own clock, Job A fires 1h37m–2h37m later (bars settled in both DST regimes), and the verified briefing publishes before 9:00pm the same evening: after-hours noise largely done, more than twelve hours before the next open. Keeping the crons UTC-fixed is still the right call — DST transitions can never skip or double a run; the only thing that moves is the label on your wall clock (an hour earlier in winter). The “briefing” is honestly an evening digest plus next-session preparation, built strictly from closed-session data — reviewed in calm, when no regular-session order could execute anyway.

Part 3

Stack decisions in detail

Each layer: what was chosen, why, what else was considered, and what you give up. The bias throughout: simplicity and low ops burden beat sophistication for a single-user system.

3.1  Frontend — Next.js App Router + TypeScript + Tailwind + shadcn/ui

Why: your core skill set; server components read the serving tables via Prisma with zero client data-fetching complexity, which suits a read-mostly app perfectly. Notes: chart components are client-side ('use client' + useEffect, or next/dynamic with ssr: false) since every serious chart library is canvas-imperative; Next.js 16 renamed middleware to proxy — the Basic Auth gate lives there. The app writes only user-state tables (watchlist, journal, review state, paper trades, weakener checkboxes) via server actions; everything pipeline-derived is read-only to it. Alternatives: none seriously — nothing beats the incumbent skill. Tradeoffs: heavy on-demand computation must go to the pipeline (or a manually triggered workflow_dispatch), never a Vercel function — the Hobby 300-second cap is a hard wall.

3.2  Serving database — Supabase free Postgres + Prisma

Why: you already know both. Prisma owns the schema and migrations; the Python pipeline INSERTs via a connection string from GitHub secrets. The feared 7-day inactivity pause is moot while the pipeline writes nightly. Connection fine print (matters in CI): three URLs, not two — the pooled transaction-mode DATABASE_URL for the app and the pipeline’s normal writes; DIRECT_URL for migrations run from your machine; and the Supavisor session-mode pooler string (IPv4, port 5432) for anything running on GitHub-hosted runners (pg_dump, CI migrations) — Supabase free-tier direct connections are IPv6-only and GitHub runners have no IPv6. Alternative: Neon free (100 CU-hours/mo, native Vercel Marketplace) is equally valid and decouples database health from pipeline health — with Prisma it is a connection-string change. Tradeoffs: a pipeline broken for 7+ days pauses the Supabase project (manual dashboard restore) — mitigated by the dead-man alert, which surfaces a broken pipeline the same evening (~9:10pm EDT / 8:10pm EST). The 500MB ceiling means full price history must not live here — hence:

3.3  History store — Parquet on Cloudflare R2, queried by DuckDB

Universe, defined once and used everywhere: US common stocks and ETFs listed on NYSE, Nasdaq, and AMEX (no OTC), ~5–6k active symbols, with delisted symbols retained for survivorship honesty. At Alpaca’s free 7+ years of history that is ~5.5k × 252 × 7 ≈ 9.7M rows ≈ 120–350MB of Parquet — DuckDB full-scans that in seconds; Stooq/ yfinance backfill can extend the archive toward 10 years later if base rates want deeper samples. Postgres adds schema ceremony for zero benefit at an append-nightly/scan-nightly access pattern. R2 is deliberately a cache, not a system of record: Alpaca history is re-downloadable, so losing the bucket costs a re-pull, not data. Adjustment correctness: raw OHLCV is stored alongside adjusted values; on any corporate action the affected symbol’s rows are rewritten across all year partitions — “immutable prior years” describes the layout’s default, never a bar to adjustment corrections. Alternatives considered: everything-in-Postgres (blows the free tier), GitHub Actions cache (7-day eviction — wrong tool), committing Parquet to the repo (~1GB/month of git bloat). Tradeoffs: one extra account and credential. R2’s free tier (10GB per research) was not independently verified — confirm at signup; Backblaze B2 is the fallback.

3.4  Pipeline runtime — Python 3.12: Polars + DuckDB + exchange_calendars + httpx

Why: the consensus 2026 pattern for exactly this workload — Polars for ingest/transform with lazy, vectorized expressions; DuckDB as the SQL engine over Parquet for “indicator state → forward return” window queries (zero-copy interop between them). Indicators are hand-rolled (~200 lines of Polars expressions for SMA/EMA/RSI/ MACD/ATR/Bollinger and friends): pandas-ta passed its announced July 2026 archival deadline, and TA-Lib’s C dependency is CI install pain you don’t need for daily-bar math. Unit-test each indicator against known reference values — this fits your TDD workflow and you own correctness. Base rates are a group_by + shift — ~50 lines of custom code, not a backtesting framework. Rerun semantics are designed, not hoped: every serving write carries a natural key with ON CONFLICT DO NOTHING (signals) or full-refresh semantics (scans), per-stage completion rows in pipeline_run let a workflow_dispatch rerun skip completed stages, and the whole night is safe to run twice. Local development: each provider adapter ships with recorded fixture responses; tests run against fixtures + a docker-compose Postgres + a local MinIO (or dev R2 bucket); CI runs the same fixtures — no test ever touches a live API key. Alternatives: TA-Lib (fine, actively maintained, use it if you’d rather not own indicator correctness); vectorbt free edition (reserve for optional parameter sweeps); pandas-ta-classic fork (longevity risk). Compute reality: the full-universe scan is minutes on a 2-vCPU runner; wall-clock is dominated by provider APIs, not math.

3.5  Scheduler — GitHub Actions, and why not Vercel

The forced departure. Verified against official docs (July 2026): Vercel Hobby caps every function — cron included — at 300 seconds with no extension; daily Hobby crons fire at an arbitrary minute within the hour; and pandas/numpy-sized Python bundles fight the packaging limits. Vercel Pro ($20/mo) buys 800s functions — worse Python ergonomics at four times the price of a VPS, rejected. A private-repo GitHub Actions workflow gives a full Python environment, a 6-hour job limit, ~2,000 free minutes/month, and workflow_dispatch for manual reruns. The minutes math is why the night is two jobs: a single job that blocked on the ~35–40-minute LLM batch would bill ~78 minutes/night ≈ 1,650/month — uncomfortably close to the free cap, and exhausting it mid-month is precisely the silent death the design fears. Splitting at batch submission (Job A ~40 min + Job B ~10 min ≈ 50 billed minutes/night ≈ 1,050/month) leaves ~45% headroom for reruns, backfills, and the weekly backup step. Cron jitter (5–30 min, community-documented) is irrelevant inside the after-close slack; the :37/:25 odd minutes dodge top-of-hour congestion. Reliability fine print: GitHub auto-disables schedules in repos with no activity for 60 days — documented for public repositories; whether it applies to private repos is unverified, so a zero-cost heartbeat step stays in the workflow and the dead-man ping remains the primary defense. Schedule events can also be delayed or occasionally dropped under platform load — workflow_dispatch is the manual recovery, and the dead-man alert is what tells you to use it. Escape hatch: a Hetzner CX22 VPS (~$5/mo, unverified — confirm at checkout) with a systemd timer, if exact timing or >6h runs ever matter; the DB stays managed so the box remains disposable.

3.6  Charts — TradingView Lightweight Charts v5, and the licensing trap avoided

Why: a purpose-built canvas candlestick engine at 59KB gzipped, Apache-2.0 with an attribution requirement satisfied by the built-in TradingView logo (on by default — leave it on and you are compliant for free). v5 brings panes (RSI/volume sub-panes), series markers and price lines — exactly the primitives the explain-why annotations need — plus a custom-primitives plugin API if annotations get fancy. Integrate via the MIT React wrapper (lightweight-charts-react-components, React 19-verified) or a ~30-line hand-rolled hook (TradingView’s own React example; zero dependency risk — arguably the right call here). The trap: TradingView Advanced Charts’ free license is companies-only, explicitly excluding personal/hobby use — legally unavailable to this project regardless of its feature appeal. Fallback: Highcharts Stock is free for personal non-commercial use (CC BY-NC) and is the turnkey option if you ever want a built-in range-selector/drawing UI; ApexCharts is no longer MIT and its SVG renderer struggles at candlestick data volumes; chartjs-chart-financial is dormant. General viz: Recharts 3 (the shadcn/ui charts foundation) for sparklines and small multiples; add tree-shaken ECharts 6 via echarts/core only when the sector heatmap becomes real.

Licensing — one rule covers everything

Every shortlisted provider licenses data for personal, non-redistributed use with local storage; several (Finnhub, Tiingo most strictly) forbid showing data to anyone else, and even paid self-serve tiers stay personal-use. US real-time redistribution is further bound by exchange agreements (SIP/OPRA) at the paid tiers — another reason v1 stays end-of-day. The always-on Basic Auth wall is the compliance mechanism. Keep the FRED attribution line and the TradingView logo rendered; no public deploy previews of data pages (Vercel’s free preview authentication covers this).

Part 5

The AI layer

Three stages, five grounding rules, one hard gate. The LLM reads and writes; it never counts, never predicts, and never publishes unverified numbers.

5.1  Extract → synthesize → verify

Cheap-fast (Haiku-class: claude-haiku-4-5) · Batch API, −50% — submitted by Job A, collected by Job B ~65 min later

C · Verification gate Deterministic, not an LLM judge — with defined tolerances, not naive string equality: numbers are parsed to values and matched within rounding tolerance (a drafted “8.2%” matches a sourced 8.24%), tickers and dates canonicalized, units normalized ($1.2B ≡ 1,200M). Every number, ticker, and date in the draft must match the source extracts or the stats table; violations are flagged inline or block publish. (LLM citation-validation runs ~38% accuracy in studies; deterministic checking is near-free.) Optional cheap-model spot-check on non-numeric claims. Parser/regex + optional Haiku pass · synchronous
~$0–0.03

5.2  The five grounding rules

  1. The pipeline computes all numbers; the LLM narrates them verbatim. Benchmark evals show top models collapsing from ~95% on simple financial lookups to near 0% on multivariate calculation — so arithmetic is simply not the model’s job here.
  2. Ground on fetched documents only: every input carries a doc ID; outside knowledge is forbidden; every claim must cite a doc ID — enforced via structured output schema (output_config.format / client.messages.parse), not via hope.
  3. Assume residual hallucination and gate it. Grounded-summarization hallucination rates are ~0.7–1.5% even for top models — low, not zero. Stage C exists to catch the residue; citations render inline in the UI so the user can always click through.
  4. Two-stage architecture caps the blast radius: the synthesis model never sees raw articles — only schema-validated extracts — bounding both context size and hallucination surface.
  5. No verbalized probabilities, ever. LLM-stated confidences are systematically overconfident; every probability on screen is a frequency computed from historical data by the pipeline (Research Report, Part 7).

Only Stage A is batched — it is where the volume is, and the Batch API is the right 50% discount for work nobody is waiting on. Local models were evaluated and rejected: a CPU-only runner is slower and measurably worse, and 2026 TCO analyses put self-hosting far above a ~$7–11/month API bill.

What the LLM never does

No arithmetic. No probabilities. No direction. No “I think this stock will…”. No content that lacks a source ID. If Stage C finds an unsourced number, the briefing publishes with that line flagged — or not at all: data beats prose, and a “briefing unavailable tonight” banner over correct scan results is an acceptable outcome; a fabricated number read before the open is not.

Part 6

Data model

A deliberate split: full-market history lives as Parquet files the pipeline scans; Postgres holds what the app serves — plus the small set of tables the user writes (watchlist, journal, review state, paper trades). Two stores, each doing the one thing it is good at.

6.1  Table definitions — the fields that carry design decisions

Part 7

Security, operations & cost

Single-user security is mostly about not inventing problems: one wall, few secrets, and monitoring designed around the one catastrophic failure mode — silence.

7.1  Authentication & the licensing wall

HTTP Basic Auth implemented in ~20 lines of Next.js middleware (proxy.ts in Next 16): credentials in env vars, constant-time comparison, HTTPS via Vercel. This is deliberately boring — for one user it is exactly sufficient, and it doubles as the data-licensing compliance mechanism (every provider requires the dashboard stay private to you). The pipeline’s revalidate route requires an Authorization: Bearer $CRON_SECRET header. Vercel’s free preview-deployment authentication stays on so preview URLs never expose data pages. Rejected: Vercel Password Protection (requires a $150/mo add-on — absurd here), Clerk (dependency overkill for n=1); the free step-up if Basic Auth’s browser popup ever grates is Auth.js with a single credentials user, or Cloudflare Access in front of a custom domain.

7.3  Monitoring, failure design, and backups

  • The catastrophic failure is a silently dead cron — stale data quietly served day after day. Defense in depth: a zero-cost heartbeat step in the workflow (the 60-day schedule auto-disable is documented for public repos; unverified for private — the heartbeat removes the question either way); workflow-failure e-mails catch crashes; and a healthchecks.io dead-man check configured on the pipeline’s own cron schedule (25 0 * * 2-6, ~45-minute grace) so weekends never false-alarm — absence of success alerts by ~9:10pm EDT (8:10pm EST), the same evening, not days later. The holiday-exit path pings success explicitly so US market holidays don’t false-alarm either.
  • Degraded > dead: each data source is isolated, so one provider outage turns into one “unavailable tonight” section, recorded in pipeline_run and shown in the dashboard footer. Ingest hard-fails if <95% of the expected universe returns, so stale data never masquerades as fresh.
  • Publish is one transaction: all serving tables are refreshed inside a single transaction (DELETE-and-INSERT of the night’s serving rows, or staged tables renamed together in the same transaction) — MVCC serves readers the previous consistent snapshot until commit, so the Desk never renders a half-written briefing and foreign keys never dangle across generations.
  • Backups follow value: Parquet is re-pullable and pipeline-derived Postgres tables are regenerable — but signal_log/signal_resolution, paper_trade, and journal_entry are irreplaceable, so a weekly pg_dump step (over the session-mode pooler — see §7.2) ships a compressed dump to R2. Restore is one command; testing it once is a Phase 1 acceptance criterion.

Ordered by expected pain. The catastrophic ones are operational silence and quiet dishonesty — both engineered against, not hoped against.

9.1  Decision changers

Three futures would legitimately reopen parts of this blueprint. Intraday trading: minute bars multiply storage ~400×, demand real-time infrastructure (SIP entitlements, a relay for client-side quotes) and a different risk posture — a different product; the Research Report’s evidence argues against wanting it. Live brokerage integration: fully available to a US resident, so the gate is earned rather than legal — order routing, credential security, and a changed risk posture arrive with it; legitimate only after the paper track record demonstrates readiness, with every friction mechanism carried over. More users than you: every data license, the auth model, and Vercel Hobby terms break simultaneously — that is a commercial product with data-redistribution contracts, not an upgrade.
