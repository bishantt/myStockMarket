Part 2

System architecture

Seven boxes, one direction of flow. Everything heavy happens in two short cloud jobs in your early evening, right after the close; the app is a thin, authenticated window onto their output. Every trigger is fixed in UTC — DST-proof by construction: when your clocks shift (you live on US Eastern), the triggers don’t move, and they sit after the close in both EDT and EST.

2.1  An evening in the life of the pipeline

All triggers are fixed in UTC; your clock is US Eastern, which observes DST. So the local times below shift by an hour between summer and winter while the UTC triggers stay put — and both regimes clear the close comfortably. The design splits the evening into two short jobs so no runner ever sits idle waiting on the LLM batch.

Why the timezone math is friendly

You live on market time, so there is no conversion to reason about — the close is 4:00pm on your own clock, Job A fires 1h37m–2h37m later (bars settled in both DST regimes), and the verified briefing publishes before 9:00pm the same evening: after-hours noise largely done, more than twelve hours before the next open. Keeping the crons UTC-fixed is still the right call — DST transitions can never skip or double a run; the only thing that moves is the label on your wall clock (an hour earlier in winter). The “briefing” is honestly an evening digest plus next-session preparation, built strictly from closed-session data — reviewed in calm, when no regular-session order could execute anyway.

Part 3

Stack decisions in detail

Each layer: what was chosen, why, what else was considered, and what you give up. The bias throughout: simplicity and low ops burden beat sophistication for a single-user system.

3.1  Frontend — Next.js App Router + TypeScript + Tailwind + shadcn/ui

Why: your core skill set; server components read the serving tables via Prisma with zero client data-fetching complexity, which suits a read-mostly app perfectly. Notes: chart components are client-side ('use client' + useEffect, or next/dynamic with ssr: false) since every serious chart library is canvas-imperative; Next.js 16 renamed middleware to proxy — the Basic Auth gate lives there. The app writes only user-state tables (watchlist, journal, review state, paper trades, weakener checkboxes) via server actions; everything pipeline-derived is read-only to it. Alternatives: none seriously — nothing beats the incumbent skill. Tradeoffs: heavy on-demand computation must go to the pipeline (or a manually triggered workflow_dispatch), never a Vercel function — the Hobby 300-second cap is a hard wall.

3.2  Serving database — Supabase free Postgres + Prisma

Why: you already know both. Prisma owns the schema and migrations; the Python pipeline INSERTs via a connection string from GitHub secrets. The feared 7-day inactivity pause is moot while the pipeline writes nightly. Connection fine print (matters in CI): three URLs, not two — the pooled transaction-mode DATABASE_URL for the app and the pipeline’s normal writes; DIRECT_URL for migrations run from your machine; and the Supavisor session-mode pooler string (IPv4, port 5432) for anything running on GitHub-hosted runners (pg_dump, CI migrations) — Supabase free-tier direct connections are IPv6-only and GitHub runners have no IPv6. Alternative: Neon free (100 CU-hours/mo, native Vercel Marketplace) is equally valid and decouples database health from pipeline health — with Prisma it is a connection-string change. Tradeoffs: a pipeline broken for 7+ days pauses the Supabase project (manual dashboard restore) — mitigated by the dead-man alert, which surfaces a broken pipeline the same evening (~9:10pm EDT / 8:10pm EST). 3.3  History store — Parquet on Cloudflare R2, queried by DuckDB

Universe, defined once and used everywhere: US common stocks and ETFs listed on NYSE, Nasdaq, and AMEX (no OTC), ~5–6k active symbols, with delisted symbols retained for survivorship honesty. At Alpaca’s free 7+ years of history that is ~5.5k × 252 × 7 ≈ 9.7M rows ≈ 120–350MB of Parquet — DuckDB full-scans that in seconds; Stooq/ yfinance backfill can extend the archive toward 10 years later if base rates want deeper samples. Postgres adds schema ceremony for zero benefit at an append-nightly/scan-nightly access pattern. R2 is deliberately a cache, not a system of record: Alpaca history is re-downloadable, so losing the bucket costs a re-pull, not data. Adjustment correctness: raw OHLCV is stored alongside adjusted values; on any corporate action the affected symbol’s rows are rewritten across all year partitions — “immutable prior years” describes the layout’s default, never a bar to adjustment corrections. Alternatives considered: everything-in-Postgres (blows the free tier), GitHub Actions cache (7-day eviction — wrong tool), committing Parquet to the repo (~1GB/month of git bloat). Tradeoffs: one extra account and credential. R2’s free tier (10GB per research) was not independently verified — confirm at signup; Backblaze B2 is the fallback.

3.4  Pipeline runtime — Python 3.12: Polars + DuckDB + exchange_calendars + httpx

Why: the consensus 2026 pattern for exactly this workload — Polars for ingest/transform with lazy, vectorized expressions; DuckDB as the SQL engine over Parquet for “indicator state → forward return” window queries (zero-copy interop between them). Indicators are hand-rolled (~200 lines of Polars expressions for SMA/EMA/RSI/ MACD/ATR/Bollinger and friends): pandas-ta passed its announced July 2026 archival deadline, and TA-Lib’s C dependency is CI install pain you don’t need for daily-bar math. Unit-test each indicator against known reference values — this fits your TDD workflow and you own correctness. Base rates are a group_by + shift — ~50 lines of custom code, not a backtesting framework. Rerun semantics are designed, not hoped: every serving write carries a natural key with ON CONFLICT DO NOTHING (signals) or full-refresh semantics (scans), per-stage completion rows in pipeline_run let a workflow_dispatch rerun skip completed stages, and the whole night is safe to run twice. Local development: each provider adapter ships with recorded fixture responses; tests run against fixtures + a docker-compose Postgres + a local MinIO (or dev R2 bucket); CI runs the same fixtures — no test ever touches a live API key. Alternatives: TA-Lib (fine, actively maintained, use it if you’d rather not own indicator correctness); vectorbt free edition (reserve for optional parameter sweeps); pandas-ta-classic fork (longevity risk). Compute reality: the full-universe scan is minutes on a 2-vCPU runner; wall-clock is dominated by provider APIs, not math.

3.5  Scheduler — GitHub Actions, and why not Vercel

The forced departure. Verified against official docs (July 2026): Vercel Hobby caps every function — cron included — at 300 seconds with no extension; daily Hobby crons fire at an arbitrary minute within the hour; and pandas/numpy-sized Python bundles fight the packaging limits. Vercel Pro ($20/mo) buys 800s functions — worse Python ergonomics at four times the price of a VPS, rejected. A private-repo GitHub Actions workflow gives a full Python environment, a 6-hour job limit, ~2,000 free minutes/month, and workflow_dispatch for manual reruns. The minutes math is why the night is two jobs: a single job that blocked on the ~35–40-minute LLM batch would bill ~78 minutes/night ≈ 1,650/month — uncomfortably close to the free cap, and exhausting it mid-month is precisely the silent death the design fears. Splitting at batch submission (Job A ~40 min + Job B ~10 min ≈ 50 billed minutes/night ≈ 1,050/month) leaves ~45% headroom for reruns, backfills, and the weekly backup step. Cron jitter (5–30 min, community-documented) is irrelevant inside the after-close slack; the :37/:25 odd minutes dodge top-of-hour congestion. Reliability fine print: GitHub auto-disables schedules in repos with no activity for 60 days — documented for public repositories; whether it applies to private repos is unverified, so a zero-cost heartbeat step stays in the workflow and the dead-man ping remains the primary defense. Schedule events can also be delayed or occasionally dropped under platform load — workflow_dispatch is the manual recovery, and the dead-man alert is what tells you to use it. Escape hatch: a Hetzner CX22 VPS (~$5/mo, unverified — confirm at checkout) with a systemd timer, if exact timing or >6h runs ever matter; the DB stays managed so the box remains disposable.

3.6  Charts — TradingView Lightweight Charts v5, and the licensing trap avoided

Why: a purpose-built canvas candlestick engine at 59KB gzipped, Apache-2.0 with an attribution requirement satisfied by the built-in TradingView logo (on by default — leave it on and you are compliant for free). v5 brings panes (RSI/volume sub-panes), series markers and price lines — exactly the primitives the explain-why annotations need — plus a custom-primitives plugin API if annotations get fancy. Integrate via the MIT React wrapper (lightweight-charts-react-components, React 19-verified) or a ~30-line hand-rolled hook (TradingView’s own React example; zero dependency risk — arguably the right call here). The trap: TradingView Advanced Charts’ free license is companies-only, explicitly excluding personal/hobby use — legally unavailable to this project regardless of its feature appeal. Fallback: Highcharts Stock is free for personal non-commercial use (CC BY-NC) and is the turnkey option if you ever want a built-in range-selector/drawing UI; ApexCharts is no longer MIT and its SVG renderer struggles at candlestick data volumes; chartjs-chart-financial is dormant. General viz: Recharts 3 (the shadcn/ui charts foundation) for sparklines and small multiples; add tree-shaken ECharts 6 via echarts/core only when the sector heatmap becomes real.

One repo, two deployables, two short workflows. The adapters/ directory is the repository pattern the data plan depends on: every provider is swappable in one module, because free tiers have a documented habit of being cut (Alpha Vantage 500→25 req/day; Finnhub’s free candles removed).
