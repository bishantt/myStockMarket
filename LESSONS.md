## The UI redesign (R0–R6, 2026-07-12)

**A guard that cannot fail is worse than no guard.** The visual-regression suite's "fonts
blocked" shot — the one meant to lock the pre-swap fallback layout — was passing while testing
absolutely nothing. Signing in first had loaded the fonts into the browser's memory cache, so the
later navigation made no network request, the route interception never fired, and the
"fonts-blocked" screenshot was quietly an ordinary one. What gave it away was that the file was
byte-identical to its neighbour. It now asserts that it actually intercepted a font request.
Whenever a test is supposed to capture an UNUSUAL state — offline, fonts blocked, an empty table,
an error boundary — assert that the unusual state actually happened.

**Measure before you take the fallback ladder.** Lighthouse came back at performance 86, and the
plan had a ready-made remedy: strip the card translucency, then remove the orbs. It would have
been easy, plausible, and completely wrong. Total Blocking Time was 20ms — the glass, the wash
and the orbs cost nothing measurable — while the page was pulling ten font files totalling 429KB
against a budget that claimed 273KB. The weight was in bytes, not pixels. Cutting two font
weights took performance from 86 to 93 and left the design intact. The plan's fallback ladder was
a reasonable guess written before anyone had measured; the measurement is what actually knows.

**A budget that measures the wrong thing is worse than no budget, because it reassures you.** The
font-budget script counted basic-latin faces and reported 273KB, and it passed. The browser
fetched 429KB, because Google splits every family into latin AND latin-ext and a real browser
takes both. The check was not wrong — it was NARROW — and it had been quietly flattering us by
about sixty percent for two phases.

**Screenshots catch what tests cannot.** Three real bugs in the redesign were invisible to a
green test suite and obvious in a picture: the Desk rendering module 02 above module 01 (the grid's
dense backfill pulling the Brief into an empty cell); the header reading "Jul 10" while the module
beneath it read "Jul 11" from the same value (a bare calendar date formatted in Eastern time,
which shifts it back a day — a trap that lib/time.ts already documented, and that I walked into
anyway); and every quantile dot in the Range Ladder squashed into an overlapping ellipse by
`preserveAspectRatio="none"`. A dot you cannot count is not a dotplot. Look at the thing.

**Removing an accident can expose a bug that was hiding behind it.** Taking the `cookies()` call
out of the Desk layout (so the theme could live on `<html>`) made the route statically
prerenderable — which is what the plan always wanted, and which the cookie read had been silently
preventing. It also meant the layout's watchlist query started running at BUILD time, on a machine
with no DATABASE_URL, and the build died. The query had never been wrapped in a try/catch, unlike
every other read in the app, and it had got away with that for six phases purely because the route
happened to be dynamic. The bug was always there. The fix just stopped hiding it.

**The plan's own rules will collide, and the collision is information.** The anti-drift grep bans
the word `transition` anywhere inside a P2 file. The motion spec explicitly permits a chevron to
rotate on a `<details>` summary. SetupCards is a P2 file with a chevron. Rather than carve an
exception into the grep — where the next reader would find it and not know why — the chevron moved
into its own component file. A coarse rule that is easy to verify beats a subtle one that is easy
to get wrong.

## The app-feel build (F0–F7, 2026-07-12)

2026-07-12 · My new navigation-timing test reported Scans as the FASTEST room in the app (14ms) when it is in fact the slowest (1342ms) → Playwright matches an accessible name by SUBSTRING, not exactly, so `getByRole("heading", { name: "Scans" })` matched the Desk's own module masthead "07 — Sectors & scans" — which is already on screen before the tab is tapped. The timer started and stopped on the page it was supposed to be navigating AWAY from → the spec now asserts the destination heading is ABSENT before every timed tap, so a locator that starts matching the origin page fails the test instead of flattering it → guard added: the absence check runs on all 8 samples of all 4 destinations, not once. The general rule, which is the one worth keeping: when a test measures the ARRIVAL of something, it must first prove that thing is not already there. `exact: true` would have fixed this one locator and left the trap armed for the next.

2026-07-12 · The new TTFB budget passed /scans/unusual-volume at 46ms — a route that does not exist yet and was answering 404 → I built the probe list from the plan's future route map (correctly — a family left out of the probe set is a gap that becomes permanent), but the check only compared latency against the budget and never looked at the status code. An empty 404 is the fastest response a server can give, so the budget was rewarding the absence of the page it was meant to be measuring → the gate now treats any non-200 as a miss, and the evidence table prints the status beside every number → guard: both the pass path and the fail path were exercised deliberately before the instrument was trusted. This is the same species as the fonts-blocked VRT shot that tested nothing (R-phase): a check that cannot fail is worse than no check, because it also produces a reassuring green tick.

2026-07-12 · There is no way to run Postgres locally on this machine → brew ships no postgresql bottle for macOS 26.3 and its source build dies in configure; the prebuilt embedded binaries link against a libz that exists only inside the macOS dyld shared cache, so there is no file to point them at; Docker Desktop's daemon is not running and cannot be started headlessly (LESSONS 2026-07-11, same wall) → the seed is verified WITHOUT a database instead: 12 unit tests assert its invariants directly (frozen mover ranks, awkward rows below the movers' cut, fills that agree with lib/paper.ts), and CI's Postgres service container runs the real `migrate deploy && db:seed` on every tag → guard: the fixture tests are negative-controlled — nudging SMCI's seeded return by 0.006 turns them red. 2026-07-12 · A stale `.next/types` broke typecheck with errors about routes I had just deleted → the same lesson as 2026-07-10, met from the other side: I created two temporary probe routes for the prefetch experiment, deleted them, and `tsc` kept type-checking generated validators that still described them → `rm -rf .next` before the next typecheck → guard: none needed beyond the existing lesson, but note the trigger is ADDING-then-removing a route, not only removing one. Any temporary route must be followed by a clean build.

2026-07-12 · A visual-regression baseline came back as a picture of Next.js's 404 page, and the bug behind it would have deleted an entire room in production → `revalidatePath(path, "layout")` DROPS THE KNOWN-PARAMS SET of any route that declares `dynamicParams = false`. Every URL in that family then 404s — permanently, until the next deploy. `/scans/[preset]` declares exactly that (a closed set of five presets; the flag is what makes /scans/garbage a real 404 rather than a 200 carrying a not-found page). The theme action had called it since P6 and the watchlist action since F1 — so the reader would have found this by changing their theme ONCE, and every scan table in the app would have been gone → the app now has NO layout-scoped revalidations anywhere: the theme applies client-side and needs none, and the watchlist names the five rooms it changes → guard added: drift rule 17 greps for `revalidatePath(..., "layout")` and fails the build. Three lessons inside the one bug: (a) a framework flag with a whole-route-tree blast radius will eventually be aimed at something you did not think about; (b) the failure was invisible in isolation and only appeared in the FULL suite, because it needed another spec to fire the action first — a bug that only exists in combination is exactly what an end-to-end suite is for; (c) it was reproduced locally in two commands (request the page → 200; call a layout revalidation; request it again → 404), which took ten minutes and ended an hour of theorising. Reproduce, then fix.

2026-07-12 · The VRT diff images have never once been uploaded → the ci.yml artifact step pointed at `app/playwright-report/`, which CI never writes: the run uses the `github` reporter, not `html`. Every visual-regression failure for six phases uploaded an empty artifact, so the one thing a pixel failure owes you — the picture — was the one thing you could not get → the step now uploads `app/test-results/`, where Playwright actually writes the expected/actual/diff triptych → guard: the very next failure produced a readable 404 screenshot, which is what identified the bug above in one look. A diagnostic that has never been exercised is not a diagnostic.

2026-07-12 · A `notFound()` inside an ISR-cached dynamic route returns HTTP 200 with the 404 page in the body → Next caches the not-found render as a static entry and serves it with an OK status. `/ticker/GARBAGE` does this today (it became ISR at F1; before that it was force-dynamic and 404'd properly). `/scans/[preset]` avoids it only because `dynamicParams = false` produces a genuine 404 → the scans route keeps the flag (and drift rule 17 protects it); the ticker's wrong status is logged for the user as [FYI] — the reader still SEES "this page could not be found", so it is a correctness wart rather than a lie on screen → guard: e2e asserts the 404 STATUS on /scans/garbage, not merely the words on the page.

2026-07-12 · The e2e that proves the PM journal saves an entry has never once proved it → it waited for a `journal-saved` marker, and that marker renders whenever `state.ok` is true — which is the INITIAL value of `useActionState`. "Saved." was on the page from the moment it loaded, before anything had been written, so the assertion passed whether or not the write happened. It only started failing when the write became genuinely slow, which is the one circumstance in which a guard that cannot fail finally does → the test asserts the TEXTAREA CLEARED instead: the form resets itself on a successful write, so an empty field cannot be true until the server action has come back ok → guard: this is the third guard-that-could-not-fail found in this build (the fonts-blocked VRT shot, the 404-scores-as-fast TTFB probe, and now this). The pattern is always the same — the assertion is true in the state the page STARTS in, so it never observed the transition it was written to observe. When writing a test, ask what the page looks like BEFORE the action: if the assertion already holds there, it is not a test.

2026-07-12 · A server action that saves one journal row took over five seconds in CI, and looked like a hang → a server action's response carries a full re-render of the page it was invoked from. On the Desk that is the entire morning — ten-odd database reads — before the client sees its result. It is a slow round trip by design, not a stuck promise, and on a contended CI Postgres it comfortably outlives Playwright's 5s default → the test's timeout now says so explicitly, with the reason → worth knowing when the Desk grows: every write from a Desk form pays for a full Desk render before the button stops saying "Saving…".

2026-07-12 · Every write on the Desk left its button stuck on "Saving…" forever — while the row landed in the database perfectly — and it had been doing so since the Desk became a cached route on 2026-07-11 → calling `revalidatePath("/")` INSIDE a server action that was invoked FROM "/" deadlocks the action's own reply: the action invalidates the very page whose fresh render it is trying to stream back as its response. The write succeeds, the browser never hears about it, and the reader concludes their journal entry was lost and writes it again → both Desk writes (the journal, and the setup cards' weakener checkboxes) now revalidate inside `after()`, which is the framework's own primitive for work that must happen once the response is finished. The reader gets their reply immediately; the cache entry is invalidated a moment later, for whoever comes next. Nothing about the freshness contract changes — only the ordering → guard: the journal e2e now asserts the FORM CLEARED (which only happens on a successful round trip) instead of a marker that was visible from page load. Isolated in one experiment: delete the revalidate line, and the same action returns in 14 seconds of test wall-clock; put it back, and twenty seconds is not enough. The bug survived six phases because the only test that could have caught it was asserting a state the page starts in.

2026-07-12 · The `feel-final` tag's CI failed on an accessibility rule that printed "nested-interactive (1 nodes)" and nothing else, on a page whose own markup contains no buttons at all → the offending element was injected by a LIBRARY. Lightweight Charts' `attributionLogo` option appends a TradingView link inside the chart container, and that container is `role="img"` — an element whose children are, by that role's own definition, part of one picture. So the licence's link was on the page, present in the DOM, painted on screen, and reachable by nobody with a keyboard. Two lessons stacked: a third-party widget can violate your accessibility rules with markup you never wrote, and `role="img"` on a container is a promise that NOTHING inside it is interactive → the logo option is off and the attribution is a real link in a caption below the chart, which is what the licence asked for in the first place ("a link on the page of your website that is available to your users") → and the gate now prints the failing node's selector and HTML. A guard that names the rule but not the node tells you that something, somewhere on a page of a hundred elements, is wrong, and leaves you to find it. That is half a guard, and I paid the other half in search time.

2026-07-12 · The service-worker test began timing out at F7 — waiting 10 seconds for the worker to take control of the page, on code that had not touched the service worker in eight phases → nothing about the worker broke. Installing a service worker means downloading its entire precache, and this release quietly made that bigger (every room became a cached, prerendered route) while the Desk now prefetches five more of them on idle. On a cold CI runner the install was still in flight when the test reloaded — and a worker that has not finished installing cannot control anything. The test was not waiting too little; it was waiting for the wrong thing, at the wrong time → it waits for `serviceWorker.ready` (an ACTIVE worker — a real state the app reaches) and THEN reloads. The general shape: when a timing test starts failing after an unrelated change, ask what the change made SLOWER before you ask what it broke — and fix it by waiting for a state, never by waiting for longer. A longer timeout is a race you have agreed to lose more slowly.

2026-07-12 · Re-enabling axe's colour-contrast rule reported 58 failures across every colour the app owns — muted, ink, the accent, the up and down chips — which reads as "the palette is broken" and is one of the most alarming things a gate can say → the palette was fine. Every page in this app fades in (`.route-fade`, opacity 0 → 1) and axe composites a foreground colour through any ancestor's opacity, so it was measuring the page WHILE IT WAS STILL ARRIVING and faithfully reporting the colours of a half-transparent screen. `text-muted` at 55% opacity really is 2.23:1; it is also not a colour any human has ever read. Wait for the fade to land and 58 failures collapse to ONE → the gate has a `settle()` helper that waits for every `.route-fade`/`.content-fade` to reach opacity 1 before axe runs → the general rule, which is not about colour: NEVER MEASURE A COLOUR, A SIZE OR A POSITION WHILE THE PAGE IS STILL ANIMATING — you will measure the animation. And note what the false alarm nearly cost: I had already written the 58 into an evidence file and asked the user to approve a palette change on the strength of it.

2026-07-12 · I surveyed contrast across both themes, found dark clean, wrote "a colour that passes is not a colour to change" into the token sheet — and CI failed on the phone Desk in dark → the survey had been run with `--project=desktop`. Midnight's grey clears 5.00:1 on a STANDARD card and 4.44:1 on a RAISED one, and the phone Desk's shelf cards are raised — so the one surface that failed was the one surface my survey could not see. I measured half the app and reported on all of it → both muted values now clear the floor against the LIGHTEST card each theme ever puts them on, and the a11y gate runs both themes × both projects (42 checks) → the lesson is not "run both projects", it is that a clean result from an instrument is only as wide as the instrument's coverage, and the coverage is the first thing to state when reporting a null finding. "No failures found" and "no failures exist" are different sentences.

2026-07-12 · A watchlist row that a test had just added VANISHED when the next button in the same row was clicked — intermittently, on a page that had passed this test for weeks → /settings was an ISR-cached route whose own server actions revalidated "/settings". Every click there runs an action that re-renders that page as its reply, and when the reply can be served from a cache entry written before the click, the reader gets back the page as it was BEFORE they touched it. The extra page-loads from a new test suite warmed the cache more aggressively and turned a rare race into a reproducible one → /settings renders on request; its actions no longer revalidate their own path; the exception is declared in three places that must agree (the page, check-routes' ALLOWLIST, drift rule 13) → the rule, which is the second time this exact shape has cost us (the first was the Desk's hanging writes): A PAGE MAY BE CACHED, OR IT MAY BE WRITTEN TO AND READ BACK IN THE SAME CLICK — NOT BOTH. NEVER REVALIDATE THE PATH YOU WERE INVOKED FROM. Every other room in this app is a reader; the one room that is a writer is the one room that cannot be cached.
